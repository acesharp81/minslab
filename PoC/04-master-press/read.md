# 04 - AI 언론동향 비서 운영·구조 안내

> 문서 기준일: 2026-07-20 (KST)
> 현재 단계: PoC 운영 검증 버전
> 상세 구조와 설치 방법: [README.md](README.md)

## 프로젝트 목적

AI 언론동향 비서는 관리자가 전 부처·기관의 검색 메타정보와 뉴스 모니터링 케이스를 설정하고, 기관별 공동 수집 후 관련도를 평가하여 기준을 통과한 기사만 카카오톡으로 알리는 PoC다.

핵심 요구사항은 다음과 같다.

- 주제·키워드·언론사별 기사 수집 및 분류
- 서버 부담을 줄이기 위한 검색 API·RSS 우선 수집
- 제목·기사 요약·원문 링크를 포함한 카카오톡 알림
- 케이스별 수집 주기와 발송 시간 설정
- 키워드 후보선정·RAG 임베딩·LLM 판정을 단계별로 결합한 관련도 평가
- 낮은 관련도 사례를 축적하여 검색 규칙 개선 자료로 활용
- 기존 홈페이지의 관리자 인증·ASGI 서비스·환경설정을 공동 사용

## 빠른 사용법

### 최초 1회 준비

1. 홈페이지 루트 `.env`에 NAVER·Kakao·Ollama·OpenRouter·Supabase 설정을 등록한다. 실제 키는 Git에 커밋하지 않는다.
2. Supabase SQL Editor에서 `PoC/04-master-press/supabase_schema.sql`을 한 번 실행한다.
3. 기존 홈페이지 서비스를 재시작하고 `/health`와 `/poc/master-press/` 접속을 확인한다.
4. 홈페이지 관리자 계정으로 로그인한다. 별도 AI 언론동향 비서 계정이나 별도 웹서비스는 없다.

### 관리자 설정

1. `관리자 페이지`에서 사용 기관을 만들고 기관명·약칭·이전 명칭·관련 인물·제외어·공식 도메인·수집 주기를 설정한다.
2. 기관 카드 안에서 케이스를 만들고 `주제 검색 사용자 프롬프트`, 포함·필수·제외·긴급 키워드와 발송 임계값을 저장한다.
3. `수신자 등록 링크`를 만들어 수신자에게 전달한다.
4. 수신자는 링크에서 카카오 로그인과 `talk_message` 권한에 동의한다. 메시지는 해당 계정의 `나와의 채팅`으로 간다.
5. 관리자 화면의 `테스트`로 수신을 확인하고 케이스에 수신자를 연결한다.
6. 공통분석 Ollama 모델과 케이스분석 OpenRouter 모델을 선택한다. 저장 즉시 다음 작업부터 적용된다.

### 일상 운영

- 수집·분석·매칭·발송·보도자료 동기화는 백그라운드 Worker가 자동 수행한다.
- `지금 수집`, `지금 동기화`, `재분석`은 기다리지 않고 즉시 실행하거나 운영자가 명시적으로 다시 검토할 때만 사용한다.
- 대시보드에서 기관/케이스를 선택해 기사, 관련도, 발송 상태, 주요 화두와 LLM 처리량을 확인한다.
- Supabase가 일시 실패하면 SQLite 운영은 계속되고 미동기화 건만 다음 예약 주기에 자동 재시도된다.

### 메뉴

- `대시보드`: 기사 목록, 공통·케이스 처리 현황, 발송 상태, 언론사·분류 통계, 주요 화두, 최근 발송 기사
- `신경망 분석`: 케이스별 실제 발송 기사만 이용한 3D 상위 주제 의미관계망
- `보도동향`: 행안부 보도자료, 관련 기사 어조별 건수, 담당부서·담당자·연락처, Markdown 원문, 양방향 연관 기사
- `관리자 페이지`: 기관·케이스·수신자·모델·개선 자료 관리

## 현재 구성

별도 포트나 별도 웹서비스를 실행하지 않는다. 기존 홈페이지 서비스가 이 폴더의 기능을 동적으로 연결한다.

```text
Nginx
  └─ 기존 Uvicorn / main.py
       ├─ 기존 홈페이지 및 관리자 인증
       ├─ /poc/master-press/              AI 언론동향 비서 화면
       ├─ /api/poc/master-press/*         AI 언론동향 비서 API
       └─ 기존 lifespan Worker
            └─ 수집 → 본문 확인 → 관련도 평가 → 발송 큐 → 카카오 전송
```

기술 구성:

- Python, 표준 ASGI 연동
- SQLite: 운영 원본, 작업 큐, 토큰, 기사 및 평가 기록
- Supabase: 케이스·기사 메타데이터·관련도 결과의 선택적 미러
- NAVER 뉴스 검색 API와 관리자 지정 RSS
- Ollama: 기사별 공통 분류·배타적 어조·요약·임베딩
- OpenRouter `google/gemma-4-26b-a4b-it:free`: 케이스별 최소 공개 증거 판정
- Kakao Login OAuth와 카카오톡 `나에게 보내기`
- `trafilatura` 기반 기사 본문 추출

Node.js, Docker, 별도 Worker 서비스는 사용하지 않는다. 기존 Uvicorn lifespan 안에서 수집 오케스트레이터, Local 공통분석 Worker 1개, OpenRouter 케이스 Worker 1개와 대기 10건 이상에서만 동작하는 Burst Worker를 독립 asyncio 작업으로 실행한다.

OpenRouter 일일 소프트 한도는 기본 800회이며 호출 성공·실패, 모델, 토큰, 처리시간을 SQLite `llm_api_calls`에 기록한다. 외부 모델에는 공개 제목, 로컬 220자 요약, 추출 근거 후보, 공개 기관·인물명, 고정 케이스 유형만 전달하고 실제 본문 근거 확인은 로컬에서 완료한다.

## 자세한 구동 원리

### 전체 처리 흐름

```text
[기관 설정]
  ↓
NAVER 뉴스 API + 기관/공용 RSS로 후보 수집
  ↓
URL 정규화 → 기관 적합성 확인 → 기사 본문 1회 저장
  ↓
로컬 Ollama 공통분석 1회 → 요약·분류·어조·엔터티·근거 확정
  ↓
소속 케이스별 키워드 후보 선정 → 독립 case_evaluation 생성
  ↓
OpenRouter Gemma 케이스 판정 → 점수·대상·근거 ID 반환
  ↓
로컬 근거 재검증 → send / low / hold 확정
  ↓
케이스별·수신자별 발송 큐 → Kakao 나에게 보내기
```

각 단계는 SQLite 작업 큐와 완료 플래그로 분리된다. 한 단계가 느리거나 외부 API가 실패해도 수집 원본과 이미 완료된 결과를 덮어쓰지 않으며, 공통분석 결과 하나를 여러 케이스가 공유한다.

### 기관 단위 기사 수집

1. 활성 기관의 기관명·약칭·이전 명칭·관련 인물을 각각 검색어로 사용한다.
2. NAVER 뉴스 검색 API와 등록된 공식 RSS/Atom을 합치고 추적 파라미터를 제거한 canonical URL로 중복을 제거한다.
3. 기관 제외어·공식 도메인·제목 및 검색 요약을 이용해 1차 후보를 줄인다.
4. `robots.txt`를 확인한 뒤 필요한 기사만 요청하고 `trafilatura`로 본문을 추출한다. 유료·로그인·차단 페이지는 우회하지 않는다.
5. 기사는 기관 단위로 한 번만 저장한다. 같은 기사를 바라보는 케이스가 늘어나도 본문 수집과 공통분석은 반복하지 않는다.

### 로컬 공통분석과 기사 임베딩

공통분석은 기사마다 한 번만 수행하며 케이스별 결과와 분리한다.

- 현재 운영 모델은 Ollama의 `qwen3:4b-instruct-2507-q4_K_M`이며 관리자에서 설치된 모델로 변경할 수 있다.
- 본문이 충분하면 제목과 본문을 사용하고, 본문이 짧거나 없으면 검색 요약문을 함께 전달한다.
- 출력은 요약, 대표 분야, 세부 분류 태그, 배타적 어조 1개(`사실전달`·`부정적`·`긍정적`), 엔터티, 상위 주제 개념, 근거로 고정한다.
- 어조는 사건의 부정성이나 재난 발생 자체가 아니라 기사 화자가 기관 또는 정책을 평가하는 태도를 기준으로 판정한다.
- 원문 입력, 시스템 프롬프트, 모델 응답과 정규화 결과를 분석보고서에 보관한다.

공통분석이 완료되면 `nomic-embed-text:latest`로 다음 입력의 768차원 기사 벡터를 한 번 생성한다.

```text
search_document: 기사 제목 + 공통 요약 + 분류 태그 + 기사 본문 앞 5,000자
```

이 벡터는 보도자료 RAG 매칭과 신경망 분석에 재사용한다. 임베딩은 생성형 답변이 아니며 Qwen이나 OpenRouter 호출을 추가하지 않는다.

### 케이스 후보 선정과 OpenRouter 판정

기관 공통분석 완료 후에만 소속 케이스로 재분배한다. 각 케이스 결과는 독립 행으로 저장되므로 한 케이스의 점수나 재분석이 다른 케이스 결과를 덮어쓰지 않는다.

1. 포함·필수·제외 키워드와 언론사 조건으로 값싼 후보 필터를 실행한다.
2. 후보만 `case_evaluation` 작업 큐에 넣는다. 키워드 점수는 후보 진단용이며 사용자 표시 유사도를 왜곡하지 않는다.
3. 현재 `google/gemma-4-26b-a4b-it:free`가 엄격한 JSON 형식으로 주제 관련성, 0~100 점수, 기관이 주된 대상인지, 근거 ID를 반환한다.
4. 모델이 선택한 근거가 실제 로컬 기사에 있는지 다시 검증하고, 부정 모니터링은 공통 어조가 `부정적`인지와 기관 직접 비판 근거를 함께 확인한다. 단순 재난 발생·중대본 가동·주의보 상향 같은 운영 사실은 부정 기사 발송에서 제외한다.

외부 모델에는 다음 최소 공개 정보만 보낸다.

- 공개 기사 제목
- 로컬에서 만든 220자 이내 요약
- 로컬에서 추출한 근거 후보와 ID
- 공개 기관명·약칭·공개 인물명
- 시스템이 정규화한 케이스 유형

기사 전체 본문과 관리자가 입력한 원문 프롬프트는 OpenRouter로 보내지 않는다. 현재 표시 퍼센트와 발송 임계값은 모두 Gemma의 실제 점수를 기준으로 한다.

```text
표시 유사도 = 검증 전 Gemma 점수(운영 사실 예외 적용 후)
발송 = 유사도 ≥ 케이스 임계값 AND 대상 근거 확인 AND 필요한 어조 근거 확인
```

- `send`: 모든 발송 조건 통과
- `low`: 분석은 완료됐지만 점수·대상·어조·제외 규칙 중 하나가 미달
- `hold`: 외부 모델 장애·한도 등으로 판정을 확정하지 못함

### 행안부 보도자료 Markdown RAG

보도자료는 기사 수집과 별도 파이프라인으로 자동 처리한다.

1. 오케스트레이터가 30초마다 실행 여부를 확인하고 기본 30분 주기로 행안부 공식 RSS의 신규 게시물을 찾는다.
2. 회차당 최대 8건의 상세 페이지에서 `#desc_pc` 본문, 담당부서, 담당자, 연락처를 추출한다.
3. `data/press_releases/mois/{연도}/{게시물ID}.md`에 front matter를 포함한 Markdown으로 저장한다.
4. 출처 URL이나 담당자 표기만 다른 동일 본문은 문서 fingerprint로 합친다.
5. Markdown을 최대 1,200자, 160자 중첩 청크로 나누고 `nomic-embed-text:latest` 768차원 벡터를 만든다.
6. 보도자료와 청크를 SQLite에 저장한 뒤 Supabase pgvector 테이블로 자동 미러링한다.

기사–보도자료 비교 후보는 다음 조건으로 제한한다.

- 같은 기관에 속함
- 기사와 보도자료 게시일 차이가 기본 ±45일 이내
- 기사 벡터와 보도자료 청크 벡터가 모두 완료됨

기사 벡터와 모든 보도자료 청크의 코사인 유사도 중 최댓값을 사용하고, 기관 공통 문구로 생기는 높은 기준선을 보정한 뒤 핵심어 근거를 합친다.

```text
원본 의미점수 = MAX(기사 벡터 ↔ 보도자료 청크별 cosine)
보정 의미점수 = clamp((원본 의미점수 - 0.68) / 0.22, 0, 1)
핵심어 점수 = 공통 불용어 제거 후 기사 제목·검색요약·본문 앞 3,000자와 보도자료 제목·요약의 단어 교집합 비율
최종 연관도 = 보정 의미점수 × 80% + 핵심어 점수 × 20%
관련 판정 = 최종 연관도 ≥ 62% AND 핵심어 점수 ≥ 15%
```

`행정안전부`, `행안부`, `정부` 같은 기관 공통어는 핵심어 계산에서 제외한다. 생성형 LLM은 이 매칭에 호출하지 않는다.

### 재처리 방지와 Supabase 자동 동기화

- 기사 공통분석은 기사별 완료 플래그, 케이스 평가는 `기사 + 케이스`, 보도자료 매칭은 `기사 + 보도자료` 복합 기본키로 보호한다.
- 매칭 대상과 발송 제외 결과도 완료 기록으로 남기므로 관련도가 낮았던 조합을 다시 계산하지 않는다.
- `ON CONFLICT DO NOTHING`과 작업 상태의 원자적 전환으로 Worker가 같은 조합을 동시에 가져가지 못하게 한다.
- 관리자 `재분석`은 자동 완료 플래그를 지우지 않고 별도 작업·결과로 저장한 뒤 반영 여부를 묻는다.

Supabase는 운영 원본이 아니라 검색·분석용 미러다.

- SQLite가 기사, 큐, 토큰, 완료 상태의 운영 원본이다.
- 성공한 보도자료와 관련 매칭에는 `supabase_synced_at`을 기록한다.
- 네트워크·Supabase 장애 시 해당 값이 비어 있는 실패분만 다음 30분 예약 주기에 자동 재시도한다.
- Supabase SQL 스키마를 한 번 적용한 이후 신규 보도자료·벡터·관련 매칭은 수동 조작 없이 들어간다.
- 관리자 `지금 동기화`는 다음 예약 시각을 기다리지 않는 즉시 실행 기능일 뿐 매번 누를 필요가 없다.

보도동향에서는 보도자료→관련 기사와 기사 카드→관련 보도자료를 양방향으로 조회하며, 보도자료별 관련 기사를 `사실전달`·`부정적`·`긍정적`으로 집계한다.

### 예약 실행과 병렬 처리

| 실행기 | 역할 |
|---|---|
| 30초 오케스트레이터 | 기관 수집 시각, 보도자료 RSS, 발송 큐, 보존 기간 정리 확인 |
| Local Worker 1개 | 재분석 → 기사 공통분석 → 기사 임베딩 → 보도자료 임베딩·매칭 순으로 로컬 작업 처리 |
| Remote Case Worker 1개 | OpenRouter 케이스 평가 처리 |
| Remote Burst Worker 1개 | 케이스 대기 10건 이상일 때만 추가 처리 |

로컬 Ollama 작업은 단일 lock으로 직렬화하여 CPU 과부하를 막고, OpenRouter 케이스 평가는 별도 Worker로 분리해 공통분석과 병렬 진행한다. 별도 systemd Worker·Docker·Node.js 서비스는 없다.

기사·케이스·발송 건수는 계속 변하므로 이 문서에 고정 수치를 기록하지 않는다. 다음 화면을 운영 기준으로 사용한다.

- 대시보드 `일일 기사 처리 현황`: 한국시간 00:00 기준 공통·케이스 처리량과 평균시간
- `LLM 처리 현황`: 현재 처리 기사 제목, Ollama/OpenRouter 모델과 호출량
- `최근 발송 기사`: `deliveries.status='sent'`인 실제 성공 건
- `보도동향`: 보도자료 임베딩·매칭 대기와 Supabase RAG 상태
- 관리자 `준비 상태`: NAVER·Kakao·암호화·Supabase·Ollama·OpenRouter 연결 여부

```bash
curl -fsS http://127.0.0.1:8000/health
curl -fsS http://127.0.0.1:8000/api/poc/master-press/dashboard
systemctl status myservice --no-pager
```

`send` 판정 수와 실제 카카오 수신 성공 수는 다르다. 최종 발송 여부는 반드시 `deliveries` 상태와 최근 발송 기사로 확인한다.

## 카카오 수신자 등록 상세 절차

1. 홈페이지 관리자 계정으로 로그인한다.
2. 왼쪽 메뉴에서 `04 - AI 언론동향 비서`을 선택한다.
3. 관리 설정에서 케이스를 생성하거나 수정한다.
4. 수신자 등록 링크를 생성하여 받을 사람에게 전달한다.
5. 수신자는 링크에서 실제 메시지를 받을 카카오 계정으로 로그인하고 메시지 권한에 동의한다.
6. 관리자 화면의 `테스트` 버튼을 누른다.
7. 수신자는 자신의 카카오톡 프로필에서 `나와의 채팅`을 열어 메시지를 확인한다.
8. 케이스에 수신자를 연결하고 수집·발송 일정과 관련도 기준을 저장한다.
9. 공개 대시보드에서 수집 기사와 점수 분포를 확인한다.
10. 관리자 개선 자료에서 낮은 점수의 원인을 주기적으로 검토한다.

Kakao Redirect URI는 다음 주소를 정확히 사용한다.

```text
https://www.minslab.kr/poc/master-press/oauth/kakao/callback
```

Redirect URI와 Webhook URL은 서로 다른 용도이므로 같은 주소로 등록하면 안 된다.

## 현재 알려진 제약

- `모든 언론사`를 직접 크롤링하는 구조는 아니다. 현재는 NAVER 뉴스 검색 결과와 등록된 RSS 범위에서 수집한다.
- NAVER 검색은 기관 메타정보별 최신 100건을 사용하며, 기관별 `회차당 검색어 수` 제한을 넘는 키워드는 해당 회차에 사용하지 않는다.
- 언론사마다 HTML 구조가 달라 기사 본문 추출 품질에 차이가 있다.
- 유료 기사, 로그인 기사, robots 차단 기사는 제목과 검색 요약문만 분석할 수 있다.
- 카카오 공식 `나에게 보내기` 방식이므로 임의의 친구 대화방이나 단체방에는 보낼 수 없다.
- 카카오 API가 성공을 반환해도 현재 관리자 화면에는 최종 `result_code`, 발송 시각, 대상 계정 확인 정보가 충분히 표시되지 않는다.
- 실패 발송은 최대 3회 재시도하지만 지수형 대기, 오류 유형별 정책, 일일 쿼터 보호는 아직 없다.
- Worker 중복 방지는 단일 프로세스 내부 lock 방식이다. Uvicorn worker를 여러 개로 늘리면 중복 실행 가능성이 있다.
- Supabase는 best-effort 미러이며 SQLite 복구용 양방향 동기화 기능은 없다.
- 저유사도 원인은 축적하지만 정답 라벨을 이용한 정량 정확도·재현율 측정은 아직 제공하지 않는다.
- 서버가 정리 시각에 중단되어 있으면 그날의 보존 기간 정리 작업이 건너뛰어질 수 있다.

## 더 개발해야 할 부분

### P0 — 운영 신뢰성

#### 카카오 발송 이력 화면 강화

- 테스트와 기사 발송의 요청 시각, 카카오 HTTP 상태, `result_code`, 재시도 횟수 표시
- `나와의 채팅으로 발송`이라는 도착 위치를 버튼 주변에 명시
- 최근 성공·실패 메시지와 오류 원인을 수신자별로 확인
- 개인정보를 늘리지 않는 범위에서 등록 계정을 식별할 수 있는 표시 제공

완료 기준: 관리자가 메시지 미수신 신고를 서버 로그 없이 화면에서 1차 판별할 수 있어야 한다.

#### 발송 재시도와 카카오 쿼터 보호

- 429, 5xx, 토큰 오류를 구분한 재시도 정책
- 1분→5분→30분 형태의 지수형 backoff
- 일일·수신자별 발송량 집계와 제한 임박 경고
- 같은 기사·케이스·수신자에 대한 중복 발송 방지 검증
- 최종 실패 큐와 관리자 수동 재전송 기능

완료 기준: 일시적인 카카오 장애가 기사 유실이나 짧은 간격의 반복 호출로 이어지지 않아야 한다.

#### 다중 프로세스용 Worker lease

- SQLite 또는 Supabase advisory 방식의 프로세스 간 lease
- Worker heartbeat와 마지막 정상 실행 시각 표시
- 서버 재시작 시 미완료 collection run과 delivery 복구
- 정리 작업을 특정 2분 구간이 아니라 `마지막 정리일` 기준으로 실행

완료 기준: Uvicorn worker 수가 늘어나거나 서버가 재시작되어도 중복 수집·중복 발송이 없어야 한다.

#### 운영 관측과 장애 알림

- 수집 API 오류, 본문 추출 실패율, Ollama 지연, 발송 실패율 집계
- 최근 실행 내역과 케이스별 소요 시간 표시
- 일정 시간 Worker 미실행 시 관리자 경고
- 민감정보를 제거한 구조화 로그와 로그 보존 정책

완료 기준: 수집·분석·발송 중 어느 단계가 멈췄는지 대시보드에서 확인할 수 있어야 한다.

### P1 — 수집 범위와 분석 품질

#### 언론사 소스 레지스트리

- 국내 언론사별 공식 RSS/API 주소 목록 관리
- 소스별 활성 여부, 최근 성공 시각, 응답 상태, 호출 간격 저장
- 언론사명 정규화와 도메인 alias 처리
- 실패가 반복되는 RSS 자동 일시중지 및 관리자 알림

완료 기준: 비용 없는 공식 수집 경로가 있는 언론사를 지속적으로 추가하면서 소스 상태를 관리할 수 있어야 한다.

#### 다중 검색어와 누락 보완

- 필수어·포함어 조합을 여러 검색어로 분리
- 검색 결과 paging과 실행별 커서 저장
- 같은 기사의 제휴 전송·URL 변형을 제목 및 내용 hash로 추가 중복 제거
- 긴급 키워드용 짧은 주기 검색과 일반 검색 분리

완료 기준: 키워드가 많은 케이스에서도 하나의 긴 검색어 때문에 관련 기사가 누락되지 않아야 한다.

#### 본문 추출 품질 개선

- 주요 언론사별 본문 selector adapter
- 날짜, 기자명, 수정 시각 등의 메타데이터 정규화
- 본문이 너무 짧거나 메뉴·댓글이 섞인 경우 품질 점수 기록
- 기사 수정 감지 후 필요한 경우 재평가

완료 기준: 본문 접근 가능 기사에 대해 본문 누락·오염 비율을 측정하고 낮출 수 있어야 한다.

#### 관련도 평가 검증 및 자동 보정

- 키워드·의미·LLM 점수별 분포와 상충 사례 표시
- 낮은 점수뿐 아니라 높은 오탐 가능성 사례도 표본화
- 설정 버전별 전송률과 평균 관련도 비교
- 주제별 임계점 및 가중치 추천
- 추천값은 자동 적용하지 않고 관리자 승인 후 반영
- 별도 검증 표본을 이용한 precision, recall, F1 측정 기능

완료 기준: 점수가 낮은 이유를 나열하는 수준을 넘어 설정 변경 전후 품질을 수치로 비교할 수 있어야 한다.

### P2 — 관리자 경험과 확장

#### 대시보드 시각화 확장

- 일·주 단위 기사 수와 평균 관련도 추이
- 언론사×주제 교차 분포
- 키워드·의미·LLM 점수의 구성 비교
- 수집→후보→본문→평가→발송 funnel
- 저점 원인 drill-down과 원문 비교 화면
- 모바일 레이아웃 및 접근성 개선

#### 수신자 생명주기 관리

- 토큰 만료 예정과 재동의 필요 상태의 명확한 표시
- 카카오 연결 끊기 Webhook 전용 endpoint와 검증
- 외부에서 앱 연결을 해제했을 때 수신자 자동 비활성화
- 등록 링크 요청 횟수 제한과 만료·사용 이력 관리

#### 설정 내보내기와 복구

- 케이스 설정 JSON 내보내기·가져오기
- SQLite 자동 백업과 복구 절차
- Supabase mirror 누락 재동기화 명령
- 스키마 버전 및 migration 체계

#### 테스트 확대

- NAVER·Kakao·Ollama 응답을 고정한 contract test
- OAuth 만료·재발급·권한 철회 시나리오
- 예약 시각, 자정, 서버 재시작, 중복 Worker 테스트
- 본문 추출 adapter fixture
- 장시간 실행과 메모리·CPU 부하 테스트
- 관리자 API의 CSRF·입력 검증·rate limit 테스트

## 권장 개발 순서

1. 카카오 발송 이력과 수신 위치 안내를 화면에 추가한다.
2. 오류 유형별 backoff와 최종 실패 관리 기능을 구현한다.
3. Worker lease, heartbeat, 재시작 복구를 구현한다.
4. 언론사 소스 레지스트리와 다중 검색어 수집을 추가한다.
5. 관련도 구성 점수와 버전별 품질 비교 화면을 만든다.
6. Webhook, 백업·복구, migration 및 보안 테스트를 보강한다.
7. 충분한 평가 표본이 쌓인 뒤 주제별 가중치·임계점 추천 기능을 적용한다.

## 운영 원칙

- 언론사 이용약관과 `robots.txt`를 우회하지 않는다.
- 검색 API와 공식 RSS를 먼저 사용하고 필요한 기사만 본문을 요청한다.
- 카카오 비공식 자동화나 개인 계정 매크로를 사용하지 않는다.
- 토큰·API 키·서비스 역할 키를 브라우저, 로그, Supabase에 노출하지 않는다.
- 자동 추천이 운영 설정을 임의로 변경하지 않도록 관리자 승인 절차를 유지한다.
- `전송 후보`, `API 수락`, `실제 사용자 확인`을 서로 다른 상태로 관리한다.
