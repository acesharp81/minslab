# 04 - AI 언론동향 비서 운영·구조 안내

> 문서 기준일: 2026-07-21 (KST)
> 현재 단계: PoC 운영 검증 버전
> 본 문서는 PoC 04의 최신 운영·구조·검증 결과를 기록한다.

## 1. 프로젝트 목적

AI 언론동향 비서는 부처·기관별 키워드로 언론 기사를 수집하고, AI가 기사 내용을 공통 분류한 뒤, 관리자가 만든 케이스별 기준에 맞는 기사만 카카오톡으로 알려주는 언론 모니터링 PoC다.

핵심 목표는 다음과 같다.

- 기관 단위로 기사를 한 번만 수집하고 여러 케이스에 재사용한다.
- 기사마다 공통분석은 한 번만 수행한다.
- 케이스별 판정은 필요한 후보만 묶어서 처리해 LLM 호출수를 줄인다.
- 관련 기사만 제목, 요약, 원문 링크와 함께 카카오톡 `나와의 채팅`으로 발송한다.
- 발송 제외·후보 제외 사유를 기록해 케이스 프롬프트와 키워드 개선 자료로 쓴다.
- 행안부 보도자료를 Markdown/RAG 형태로 수집해 기사와 양방향으로 연결한다.
- 기존 홈페이지의 ASGI 서비스, 관리자 인증, 루트 `.env`를 그대로 사용한다.

별도 Node.js, Docker, 별도 포트, 별도 웹서비스는 사용하지 않는다.

## 2. 빠른 사용법

### 관리자 기준 기본 흐름

1. 기존 홈페이지 관리자 계정으로 로그인한다.
2. `관리자 페이지`에서 사용 기관을 만든다.
   - 기관명
   - 약칭
   - 이전 명칭
   - 관련 인물
   - 제외 키워드
   - 공식 도메인/RSS
   - 수집 주기
3. 기관 아래에 케이스를 만든다.
   - 케이스명
   - 주제 검색 사용자 프롬프트
   - 포함 키워드
   - 필수 키워드
   - 제외 키워드
   - 긴급 키워드
   - 발송 임계값
   - 수신자
4. 구독 신청 또는 수동 등록 링크로 카카오 수신자를 등록한다.
5. 관리자가 회원 관리에서 케이스별 구독 신청을 승인한다.
6. Worker가 자동으로 수집, 분석, 판정, 발송을 수행한다.
7. 대시보드에서 처리 상태와 발송 결과를 확인한다.

### 일반 수신자 구독 신청 흐름

1. `구독 신청` 페이지에서 이름과 기관을 선택한다.
2. 활성 케이스 목록에서 수신할 케이스를 선택한다.
3. 구독 요청 후 표시되는 카카오 등록 URL을 연다.
4. 카카오 로그인과 `talk_message` 권한에 동의한다.
5. 관리자가 승인하면 해당 케이스의 발송 대상이 된다.
6. 관리자가 승인 해제를 누르면 먼저 카카오 안내 문자가 발송되고, 성공 후 케이스 수신 연결이 해제된다.

신청자 이름은 공개 신청 목록에서 `김철수 → 김*수`처럼 마스킹한다.

## 3. 화면 구성

- `구독 신청`: 수신 신청, 카카오 등록 URL 확인, 신청 상태 확인
- `대시보드`: 기사 분석 기록, 처리 현황, 주요 화두, 최근 발송 기사, 통계
- `신경망 분석`: 발송된 기사만 사용한 케이스별 3D 의미관계망
- `보도동향`: 행안부 보도자료 목록, 제목·담당부서·담당자 검색, 관련 기사 카운트, Markdown 원문, 관련 기사
- `관리자 페이지`: 회원 관리, 기관·케이스·모델·구독 승인·개선 자료 관리

보도동향에는 브라우저 로컬 쿠키 기반 즐겨찾기가 있다. 보도자료 제목 앞의 `☆`를 누르면 `★`로 바뀌고, 최대 3개까지 목록 최상단에 고정된다. 이 값은 서버 DB에 저장하지 않는다.

대시보드 기사 카드의 `+케이스` 확장 목록에서는 상태 배지에 마우스를 올리면 후보 제외, 발송 제외, 판정 실패 등의 사유가 커스텀 툴팁으로 표시된다.

## 4. 시스템 구성

```text
Nginx
  └─ 기존 Uvicorn / main.py
       ├─ 기존 홈페이지 및 관리자 인증
       ├─ /poc/master-press/              AI 언론동향 비서 화면
       ├─ /api/poc/master-press/*         AI 언론동향 비서 API
       └─ lifespan Worker
            ├─ 기관 기사 수집
            ├─ Groq 공통분석
            ├─ Ollama 임베딩
            ├─ OpenRouter 케이스 판정
            ├─ 보도자료 수집·임베딩·매칭
            └─ Kakao 발송
```

주요 구성요소는 다음과 같다.

| 구성 | 역할 |
|---|---|
| SQLite | 운영 원본 DB, 큐, 토큰, 기사, 분석, 발송 상태 저장 |
| Supabase | 보도자료·기사 메타데이터·벡터·매칭 결과 미러 |
| NAVER News API | 기관 키워드 기반 기사 검색 |
| RSS/Atom | 기관 또는 언론사 공식 피드 수집 |
| Groq | 기사 공통분석 LLM |
| OpenRouter | 케이스별 의미 판정 LLM |
| Ollama | 로컬 임베딩 전용 |
| Kakao Login/API | OAuth 수신자 등록, `나에게 보내기` 발송 |
| trafilatura | 기사 본문 추출 |

현재 기본 모델 방향은 다음과 같다.

- 공통분석: Groq `llama-3.1-8b-instant`
- 케이스 판정: OpenRouter `google/gemma-4-26b-a4b-it:free`
- 임베딩: Ollama `nomic-embed-text:latest`

관리자 화면에서 Groq, OpenRouter, Ollama 임베딩 모델을 선택할 수 있으며 저장 즉시 다음 작업부터 적용된다.

## 5. 기사 처리 파이프라인

전체 처리 흐름은 아래 순서다.

```text
[기관 설정]
  ↓
기관 키워드 기반 NAVER/RSS 수집
  ↓
URL 정규화·중복 제거·기관 후보 필터
  ↓
기사 본문 수집 및 articles 저장
  ↓
article_analysis_jobs 생성
  ↓
Groq 공통분석 1회
  ↓
article_embeddings 생성
  ↓
기관 소속 케이스 후보 게이트
  ↓
case_evaluations 생성
  ↓
case_evaluation_jobs 배치 처리
  ↓
OpenRouter 케이스 판정
  ↓
하이브리드 점수·하드 게이트 최종 결정
  ↓
deliveries 큐 생성
  ↓
Kakao 나에게 보내기
```

### 5.1 기관 단위 기사 수집

수집은 케이스가 아니라 기관을 기준으로 한다. 예를 들어 행정안전부 기관에는 다음 메타정보가 들어간다.

- 행정안전부
- 행안부
- 행자부
- 장관·차관 등 공개 인물명
- 제외 키워드
- 공식 도메인
- 공식 RSS

Worker는 활성 기관의 메타정보로 검색어를 만들고 NAVER 뉴스 API와 RSS/Atom을 조회한다. URL은 canonical URL로 정규화하고 추적 파라미터를 제거한다. 같은 기사는 한 번만 `articles`에 저장한다.

본문 수집은 서버 부담을 줄이기 위해 다음 원칙을 따른다.

- `robots.txt`를 확인한다.
- 도메인별 최소 요청 간격을 둔다.
- 응답 크기를 제한한다.
- 유료·로그인·차단 기사는 우회하지 않는다.
- 본문이 부족하면 제목과 검색 요약을 보조 입력으로 사용한다.

중요한 구조적 원칙은 “기관 기사 1건 = 수집 1회”다. 케이스가 2개든 20개든 같은 기사 본문을 케이스마다 다시 가져오지 않는다.

### 5.2 공통분석

공통분석은 기사 1건당 한 번만 수행한다. 결과는 모든 케이스가 공유한다.

Groq 공통분석의 산출물은 다음과 같다.

- 기사 요약
- 대표 기사 유형
- 주제별 분류 태그
- 배타적 어조 1개
  - `사실전달`
  - `부정적`
  - `긍정적`
- 주요 엔터티
- 상위 주제 개념
- 근거 문장
- 분석보고서

어조 분류는 사건 자체가 나쁜지 여부가 아니라 기사 화자의 태도를 본다. 예를 들어 재난 발생, 경계 상향, 중대본 가동 자체는 기본적으로 사실전달이며, 기관 대응에 대한 비판·책임 추궁·시정 요구가 기사 핵심일 때 부정적으로 본다.

공통분석 결과는 `article_analyses`에 저장되고 `article_processing_flags`로 완료 여부를 관리한다. 이미 완료된 기사는 프로그램 수정이나 서버 재시작만으로 자동 재분석하지 않는다.

### 5.3 기사 임베딩

공통분석이 완료되면 Ollama 임베딩 Worker가 기사 벡터를 생성한다.

```text
search_document: 제목 + 공통 요약 + 분류 태그 + 본문 앞 5,000자
```

이 벡터는 다음 작업에 재사용된다.

- 케이스 프롬프트 벡터와의 유사도 계산
- 보도자료 RAG 매칭
- 신경망 분석

임베딩은 생성형 LLM 호출이 아니다. Groq/OpenRouter 호출량을 늘리지 않는다.

### 5.4 케이스 후보 게이트

공통분석과 기사 임베딩이 완료된 뒤에만 기관 소속 케이스로 분배한다. 이 단계의 목적은 OpenRouter에 보낼 필요가 없는 케이스를 먼저 제거하는 것이다.

현재 후보 게이트는 다음 순서로 동작한다.

1. 언론사 포함/제외 조건을 확인한다.
2. 케이스 제외 키워드가 기사 제목·요약·본문에 있으면 후보 제외한다.
3. 필수 키워드가 있는 케이스는 필수 키워드가 모두 맞아야 한다.
4. 포함 키워드가 있는 케이스는 최소 하나 이상 맞아야 한다.
5. 다만 포함 키워드가 직접 맞지 않아도 의미점수가 매우 높으면 예외적으로 통과할 수 있다.
6. 부정 모니터링처럼 포함·필수 키워드가 비어 있고 프롬프트가 부정 보도를 요구하는 케이스는 공통분석 어조 또는 부정 신호가 있어야 한다.
7. 최종적으로 키워드 후보이거나 보정 벡터 점수가 기준 이상이면 케이스 판정 큐로 들어간다.

기본 벡터 후보 기준은 `65`다. 과거에는 `35`라 기관 문맥만 비슷해도 여러 케이스가 후보가 되는 문제가 있었다. 현재는 필수·포함 조건과 부정 신호 게이트를 먼저 적용해 불필요한 케이스 판정 LLM 호출을 줄인다.

후보 제외된 경우에도 `case_evaluations`에 `excluded` 상태로 남긴다. 새로 생성되는 후보 제외 건은 `required_terms_missing`, `include_terms_missing`, `exclude_terms_matched`, `negative_signal_missing` 같은 사유를 기록한다.

### 5.5 케이스 임베딩

케이스별 주제 검색 사용자 프롬프트와 포함·필수 키워드를 기반으로 `case_embeddings`를 만든다.

- 케이스 버전별로 한 번만 생성한다.
- 프롬프트가 바뀌면 케이스 버전이 올라가고 새 벡터를 만든다.
- 제외 조건 문장은 retrieval text에서 최대한 제거해 벡터가 반대로 끌려가지 않게 한다.
- 기사 벡터와 케이스 벡터의 코사인 유사도를 계산한다.
- 케이스별 기사 분포의 q10/q50/q90으로 원시 유사도를 보정한다.

보정 벡터 점수는 OpenRouter 호출 전 후보 선정과 최종 하이브리드 점수에 모두 사용된다.

### 5.6 OpenRouter 케이스 판정

OpenRouter 판정은 “기사 1건 × 케이스 1개” 호출이 아니라, 같은 기사에 대해 대기 중인 케이스를 최대 10개까지 한 요청으로 묶는 구조다.

배치 선택은 가장 오래된 1건만 보는 방식이 아니라, 현재 대기열에서 같은 기사로 가장 많이 묶을 수 있는 그룹을 우선한다. 이렇게 해야 케이스가 10~20개로 늘어나도 호출수가 기사 수에 가깝게 유지된다.

OpenRouter에는 다음 정보가 전달된다.

- 기사 제목
- 공통분석 요약
- 본문에서 선별한 공유 근거 문장
- 기관명·공개 인물명 등 공개 타깃
- 케이스명
- 주제 검색 사용자 프롬프트
- 포함·필수·제외 키워드

기사 원문 전체를 OpenRouter에 보내지 않는다. 공통분석 단계에서 만든 요약과 선별 근거를 사용한다.

시스템 프롬프트의 핵심 규칙은 다음과 같다.

- 케이스들을 서로 비교하지 말고 독립적으로 판정한다.
- 기관이 언급됐다는 이유만으로 높은 점수를 주지 않는다.
- 단순 언급, 주변 사례, 저작권 문구는 제외한다.
- `score`는 고정 배점이 아니라 0.0~100.0 연속 유사도로 판단한다.
- 5점 단위 대표값만 습관적으로 고르지 않도록 지시한다.
- 모든 `case_id`를 정확히 한 번씩 반환한다.

결과 일부가 누락되면 성공한 케이스는 확정하고 누락된 케이스만 다시 처리한다.

### 5.7 최종 유사도와 발송 결정

최종 표출 유사도는 하이브리드 점수다.

```text
최종 유사도 = 보정 벡터 점수 × semantic_weight + LLM 점수 × llm_weight
```

현재 일반 구성은 50:50이다.

키워드 점수는 후보 선정과 진단용으로 쓰지만, 사용자에게 보이는 최종 퍼센트에는 직접 섞지 않는다. 사용자의 원래 의도인 “LLM이 사용자 프롬프트와 기사 핵심 내용의 일치도를 판단하고, 벡터 유사도로 보완하는 구조”를 유지하기 위해서다.

최종점수가 발송 임계값을 넘더라도 다음 조건을 통과하지 못하면 발송하지 않는다.

- LLM이 관련 기사라고 판단했는가
- 필수 주제를 만족했는가
- 케이스 대상이 기사 핵심인가
- 부정·긍정 등 요구 어조가 맞는가
- 근거 문장이 실제 기사 내용에 존재하는가
- 제외 조건을 위반하지 않는가
- 운영 사실을 비판 기사로 오판하지 않았는가

결정 상태는 다음과 같다.

| 상태 | 의미 |
|---|---|
| `send` | 발송 조건 충족 |
| `low` | 분석 완료, 발송 조건 미충족 |
| `excluded` | LLM 판정 전 후보 제외 |
| `pending` | 판정 대기 |
| `processing` | 판정 중 |
| `failed` | 판정 실패 |
| `hold` | 외부 모델 장애·한도 등으로 확정 보류 |

### 5.8 발송 큐와 카카오 전송

`send`로 확정되면 케이스에 연결된 수신자별로 `deliveries` 큐가 생성된다.

카카오 메시지는 다음 형식이다.

```text
[기관/분류/어조 태그]
기사 제목
요약
원문 보기 링크
```

카카오 API 제한 때문에 기본 텍스트 템플릿 200자 제한을 지킨다. 발송은 카카오 계정의 일반 친구방이나 단체방이 아니라 등록한 계정의 `나와의 채팅`으로 간다.

발송 실패는 재시도 상태로 남기며, 성공·실패·대기 상태는 대시보드와 관리자 화면에서 확인한다.

## 6. LLM 사용량 최소화를 위한 조치

이 프로젝트는 케이스가 늘어날수록 LLM 호출이 폭증할 수 있으므로, 초기 설계부터 호출수 절감을 핵심 구조로 둔다.

### 6.1 기사 수집과 공통분석을 기관 단위로 공유

과거의 위험한 구조는 “케이스마다 같은 기사를 다시 분석”하는 방식이었다. 현재는 다음처럼 나뉜다.

```text
기사 1건
  ├─ 본문 수집 1회
  ├─ Groq 공통분석 1회
  ├─ Ollama 기사 임베딩 1회
  └─ 여러 케이스가 결과 공유
```

따라서 동일 기관에 케이스가 늘어나도 공통분석 호출은 기사 수에 비례하고, 케이스 수에 직접 비례하지 않는다.

### 6.2 OpenRouter 호출 전 로컬 후보 게이트

OpenRouter를 부르기 전에 로컬에서 다음을 먼저 처리한다.

- 제외 키워드 포함 여부
- 필수 키워드 충족 여부
- 포함 키워드 충족 여부
- 언론사 조건
- 부정 케이스의 부정 신호 존재 여부
- 보정 벡터 후보 기준 통과 여부

명백히 조건을 못 맞춘 케이스는 `excluded`로 저장하고 OpenRouter에 보내지 않는다.

### 6.3 벡터 유사도는 로컬 임베딩 재사용

기사 벡터와 케이스 벡터는 한 번 만들어 재사용한다. 이 벡터는 다음 용도에 함께 사용된다.

- 케이스 후보 선정
- 최종 하이브리드 점수
- 보도자료 매칭
- 신경망 분석

새로운 분석 화면을 만든다고 해서 매번 LLM을 다시 호출하지 않는다.

### 6.4 케이스 판정 배치

동일 기사에 대기 중인 케이스를 최대 10개까지 OpenRouter 한 번으로 처리한다.

```text
나쁜 구조: 기사 1건 × 케이스 10개 = OpenRouter 10회
현재 구조: 기사 1건 × 케이스 최대 10개 = OpenRouter 1회
```

대기열도 “가장 오래된 1건”이 아니라 “같은 기사로 가장 많이 묶을 수 있는 그룹”을 우선해 작은 배치가 반복되는 일을 줄인다.

### 6.5 실패·429 재시도 제한

OpenRouter 429나 외부 오류가 발생하면 짧은 간격으로 반복 호출하지 않는다.

- 일반 429는 일정 시간 뒤 재시도한다.
- 일일 soft limit은 다음 기준 시점까지 defer한다.
- attempts 제한을 두어 무한 pending 반복을 막는다.
- 실패한 호출도 `llm_api_calls`에 기록해 대시보드에서 확인할 수 있다.

### 6.6 보도자료 RAG는 생성형 LLM을 추가로 쓰지 않음

기사–보도자료 매칭은 저장된 기사 임베딩, 보도자료 청크 임베딩, 공통분석 메타데이터, 희소 키워드 점수를 사용한다. 생성형 LLM을 다시 호출하지 않는다.

### 6.7 재분석은 자동 덮어쓰기 금지

관리자가 `분석과정`에서 재분석을 요청하면 별도 재분석 작업으로 들어간다. 결과가 나오면 관리자에게 반영 여부를 묻고, `예`를 눌러야 기존 결과를 대체한다.

자동 Worker는 이미 완료된 기사·케이스 결과를 프로그램 수정만으로 다시 분석하지 않는다.

## 7. 보도자료 RAG 파이프라인

현재 PoC는 행정안전부 보도자료를 우선 지원한다.

```text
행안부 RSS 신규 탐지
  ↓
상세 페이지 크롤링
  ↓
본문·담당부서·담당자·전화번호 추출
  ↓
Markdown 저장
  ↓
청크 분할
  ↓
Ollama 임베딩
  ↓
기사 임베딩과 매칭
  ↓
SQLite 저장 및 Supabase 미러
  ↓
보도동향/대시보드에서 양방향 조회
```

보도자료는 `data/press_releases/mois/{연도}/{게시물ID}.md` 형태로 저장한다.

매칭 후보는 같은 기관과 게시일 차이 기본 ±45일 범위로 제한한다. 연관도는 다음 요소를 합친다.

- 기사 벡터와 보도자료 청크 벡터의 의미 유사도
- 제목·요약·주제 개념 기반 희소 키워드 점수
- 상위 주제 개념 일치 여부
- 기사일과 보도자료일의 시간 거리

보도동향 화면에서는 보도자료별 관련 기사 수를 어조별로 보여준다.

- 사실전달
- 부정적
- 긍정적

보도자료 제목을 펼치면 요약, 담당부서, 담당자, 연락처, 관련 기사, 전체 Markdown 보기를 확인할 수 있다.

## 8. 신경망 분석과 주요 화두

대시보드의 주요 화두는 발송 대상 기사에서 명사와 고유명사를 중심으로 Top 10을 뽑는다. 케이스를 선택하면 케이스 단위, 케이스가 없고 기관만 선택되면 기관 단위로 집계한다.

신경망 분석은 발송된 문자 메시지 기준 기사만 노드로 사용한다. 기사 간 관계는 기관 문맥이 아니라 상위 주제 개념과 기사 내용의 유사도를 중심으로 계산한다.

목표는 다음과 같다.

- 관련 있는 화두는 가깝게 묶는다.
- 관련 없는 화두는 떨어뜨린다.
- 3D 회전과 원근감으로 멀리 있는 노드를 작고 옅게 보이게 한다.
- 케이스별 발송 기사 흐름을 시각적으로 탐색한다.

## 9. 가입 신청과 수신자 관리

가입 신청은 공개 페이지에서 진행하지만 승인은 관리자 페이지에서 한다.

상태 흐름은 다음과 같다.

```text
신청(requested)
  ↓
카카오 수신등록완료(kakao_registered)
  ↓
케이스별 승인/반려
  ↓
전체 승인(approved) 또는 일부 승인(partial) 또는 반려(rejected)
  ↓
관리자 해제 시 revoked
```

승인 시 해당 신청자의 카카오 recipient가 케이스 수신자 목록에 추가된다. 해제 시에는 먼저 카카오 안내 메시지를 보내고, 성공하면 `case_recipients` 연결을 삭제한다.

수신자 토큰은 암호화해 SQLite에 저장한다. access token은 만료 전에 refresh token으로 자동 갱신한다. 관리자 화면에는 오해를 줄이기 위해 토큰 만료 시각 대신 연결 성공/실패 상태를 표시한다.

## 10. 재처리 방지와 상태 관리

중복 분석 방지를 위해 단계별 flag와 복합키를 사용한다.

| 데이터 | 중복 방지 기준 |
|---|---|
| 기사 | canonical URL |
| 공통분석 | article_processing_flags.article_id |
| 기사 임베딩 | article_analysis_id + embedding_model |
| 케이스 평가 | article_id + case_id + article_case_processing_flags |
| 케이스 표시순서 | cases.sort_order |
| 케이스 임베딩 | case_id + case_version + embedding_model |
| 발송 | article_id + case_id + recipient_id |
| 보도자료 | canonical URL + document_fingerprint |
| 기사–보도자료 매칭 | article_id + press_release_id + matcher_version |

서버 재시작 시 processing 상태였던 작업은 안전하게 pending으로 복구한다. 완료된 작업은 다시 pending으로 돌리지 않는다.

## 11. 대시보드 처리 현황

대시보드는 한국시간 00:00 기준으로 일일 처리량을 계산한다.

표시 항목은 다음과 같다.

- 분석 기사
- 공통분석 대기/완료/오류
- 임베딩 대기/완료/오류
- 케이스별 판정 대기/완료/오류
- 발송 대기/완료/오류
- 평균 처리 속도
- 총 처리 건수
- Groq/OpenRouter/Ollama 호출 수, 모델명, 플랫폼
- 현재 처리 중인 기사 제목

자동 갱신 주기는 10초, 30초, 1분, 안함 중 선택할 수 있다. 갱신은 화면 표시만 새로고침하며 Worker 실행 주기를 바꾸지 않는다.

### 11.1 2026-07-21 운영 반영 결과

이번 운영 검증에서 반영한 핵심 변경은 다음과 같다.

- 기사 카드 상단 태그와 하단 태그가 중복 표출되던 문제를 정리해 하단 중복 태그를 제거했다.
- 보도동향에는 제목, 담당부서, 담당자명 검색어 입력칸을 추가했다.
- 후보 제외와 발송 제외 사유는 내부 코드 대신 일반 사용자가 이해할 수 있는 짧은 문장으로 표출한다.
- 기사별 처리 요약은 `케이스 n개 분석: n 처리전, n 후보제외, n 발송제외, n 발송완료` 형식으로 표기한다.
- 실제 발송된 기사는 유사도와 관련 보도자료 버튼 아래에 붉은색 `발송 n건`으로 표시한다.
- 관련 보도자료 최소 유사도 기준은 기본 65%이며, 관리자 화면에서 조정할 수 있다.
- 케이스 발송 기준은 기본 70%로 맞췄고, 기존 케이스도 일괄 70%로 보정했다.
- 최종 유사도는 기본적으로 벡터 25%, LLM 75% 비율로 계산한다. LLM 판독이 95%라면 벡터 점수가 낮아도 발송 기준을 통과할 수 있다.
- 기사 단위 재분석 버튼은 관련 보도자료 아래, `+ 케이스` 왼쪽에 배치했다. 실행 시 해당 기사 케이스 판정이 다시 대기 상태로 들어가고 최신 결과로 덮어쓴다. 조건을 만족하면 카카오 발송까지 이어진다.
- 관리 설정의 회원 관리는 최상단으로 올렸고, 동일 카카오 등록 정보를 가진 구독자는 하나의 회원으로 묶어 보이게 했다.
- 회원 목록은 접힌 상태에서 이름과 `n 케이스 승인요청, n 케이스 승인완료`를 표시하고, 펼치면 PC 기준 한 줄 3개 카드 비율로 구독 케이스를 관리한다.
- 관리자는 회원별 체크박스를 풀거나 다시 선택한 뒤 적용 버튼으로 구독 케이스를 조정할 수 있다.
- 관리자 해제 시 구독 해지 알림 메시지를 먼저 보내고, 성공하면 카카오 수신 등록 연결을 해제한다.
- 구독 신청 화면은 `이름·기관 선택 → 구독 대상 선택 → 카카오 전송` 흐름으로 좌에서 우로 입력하도록 정리했다.
- 구독 신청 화면에는 개인정보를 수집하지 않는다는 안내와, 구독 변경 시 전체 구독 대상을 다시 체크해야 한다는 안내를 넣었다.
- 신청 기록은 승인 또는 반려 후 6시간 뒤 자동 삭제한다.
- 메인 홈 서비스 세부 상태에는 로컬 LLM 호출과 Ollama 임베딩 호출을 구분해 표시한다.
- 오늘 평균 처리 속도는 개발 과정 왜곡을 제거하기 위해 `2026-07-21T15:00:09+09:00` 이후 기준으로 한 번 리셋했다.
- 오류 표시는 `현재 오류 대기 건수(누적: n건)` 형태로 표기한다.
- `worker_restarted`는 서비스 재시작 복구 표시이므로 사용자-facing 오류 집계에서 제외한다.
- 케이스 판정 배치 응답에서 일부 case_id 결과가 빠지는 `batch_result_missing`은 단건 판정으로 즉시 fallback한다.
- 오류 누적값은 `2026-07-21T16:07:39+09:00`에 한 번 리셋했고, 이후부터는 작업 재시도 횟수가 아니라 실제 API 호출 실패 건수만 일 단위로 누적 표출한다.

오류 누적 기준은 다음과 같다.

| 화면 단계 | 현재 오류 | 누적 오류 |
|---|---|---|
| 공통분석 | 현재 대기/실패 작업 중 실제 오류 상태 | Groq `common` API 실패 호출 수 |
| 임베딩 | 현재 실패한 임베딩 작업 수 | Ollama `embedding` 실패 호출 수 |
| 케이스별 판정 | 현재 대기/실패 작업 중 실제 오류 상태 | OpenRouter `case` API 실패 호출 수 |
| 발송 | 현재 retry/failed 발송 건수 | 카카오 발송 재시도·실패 누적 |

조사 당시 케이스별 판정 누적 414~424건은 현재 414건이 막힌 상태가 아니라, OpenRouter 429와 배치 누락으로 인해 케이스 작업 재시도가 누적된 값이었다. 수정 후 대시보드 API 기준 공통분석, 임베딩, 케이스별 판정의 현재 오류와 누적 오류는 모두 0건으로 확인했다.

## 12. 환경설정

실제 비밀값은 루트 `.env`에만 둔다. Git에는 커밋하지 않는다.

| 영역 | 주요 값 |
|---|---|
| NAVER | `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET` |
| Kakao | `KAKAO_REST_API_KEY`, `KAKAO_REDIRECT_URI`, 토큰 암호화 키 |
| Groq | `GROQ_API_KEY`, `MASTER_PRESS_GROQ_COMMON_MODEL`, 일일/분당 soft limit |
| OpenRouter | `OPENROUTER_API_MYKEY`, `MASTER_PRESS_OPENROUTER_CASE_MODEL`, 일일 soft limit |
| Ollama | `OLLAMA_BASE_URL`, `MASTER_PRESS_EMBEDDING_MODEL` |
| Supabase | `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` |
| 보도자료 | RSS URL, 동기화 주기, 매칭 임계값, 매칭 기간 |

Supabase는 `supabase_schema.sql`을 SQL Editor에서 한 번 적용해야 한다. 스키마가 없어도 SQLite 기반 수집·분석·화면은 동작하지만 Supabase 미러 상태는 `schema_required`로 표시될 수 있다.

## 14. 현재 알려진 제약

- `모든 언론사` 직접 크롤링 구조는 아니다. 현재는 NAVER 뉴스 검색과 등록 RSS 중심이다.
- 언론사별 HTML 구조가 달라 본문 추출 품질 차이가 있다.
- 유료·로그인·robots 차단 기사는 본문 없이 제목·요약 중심으로 처리될 수 있다.
- 카카오 공식 API 정책상 임의 친구방/단체방 발송은 지원하지 않는다.
- OpenRouter 무료 모델은 공급자 상태와 일일 한도에 영향을 받는다.
- Worker 중복 방지는 기본적으로 단일 Uvicorn 프로세스 내부 lock에 의존한다.
- Supabase는 운영 원본이 아니라 미러다. 원본 복구는 SQLite 백업을 기준으로 해야 한다.
- 후보제외 사유는 최신 게이트 이후 생성된 건부터 상세 사유가 남는다. 과거 excluded 기록은 일반 안내로 보일 수 있다.

## 17. 운영 원칙

- 수집은 기관 단위로 한다.
- 공통분석은 기사 단위로 한 번만 한다.
- 케이스 판정은 후보 게이트를 통과한 것만 한다.
- 발송은 케이스별로 독립 결정한다.
- 낮은 점수와 제외 사유도 데이터로 남긴다.
- Supabase는 미러이며 SQLite가 운영 원본이다.
- 모델 변경은 다음 작업부터 적용하고 과거 결과를 자동으로 덮지 않는다.
- 재분석 결과는 관리자 확인 후 반영한다.
- API 키와 토큰은 문서나 Git에 기록하지 않는다.
